Data Flow & Event Contracts

Every interaction follows a publish → subscribe pattern through the MessageBus

Core Event Flow

Startup & Auth Flow
App Start → AuthModule
loads cached token → AuthReadyEvent → APIClient activated
↓ if no cached token
UI shows login prompt → AuthModule
starts OAuth flow → AuthReadyEvent
Search & Play Flow
User types search → SearchRequestEvent → APIClient
calls ytmusicapi
↓
SearchResultsEvent → UI renders results
↓ user selects track
PlayRequestEvent
(track_id) → Player fetches
stream URL, starts mpv → TrackChangedEvent → UI updates
Now Playing

Event Catalog

Event Publisher Subscriber Payload
AuthReadyEvent AuthModule API, UI —
AuthErrorEvent AuthModule UI error_msg
SearchRequestEvent UI APIClient query, filter
SearchResultsEvent APIClient UI results: list[Track]
PlayRequestEvent UI Player track_id, context
TrackChangedEvent Player UI, API Track
PlaybackStateEvent Player UI state, position_s
QueueUpdatedEvent Player UI queue: list[Track]
LibraryUpdateEvent APIClient UI tracks/albums/artists
ErrorEvent Any UI source, message
MessageBus contract: All events are typed dataclasses. bus.publish(event) is async. Subscribers register with bus.subscribe(EventType, handler). Order of handlers is deterministic. No module imports another directly — all coupling is through events.
